Spring学习

Spring概述

​ spring是一个开源的开发框架，简化了企业级开发，是java开发中不可替代的框架容器，spring的核心思想是IOC、DI、AOP。

Ioc即控制反转，将bean的创建交给第三方完成，将设计好的对象交给容器来控制，IoC对编程带来的最大改变不是从代码上，而是从思想上，发生了“主从换位”的变化。应用程序原本是老大，要获取什么资源都是主动出击，但是在IoC/DI思想中，应用程序就变成被动的了，被动的等待IoC容器来创建并注入它所需要的资源了。

​ Spring所倡导的开发方式就是如此，所有的类都会在spring容器中登记，告诉spring你是个什么东西，你需要什么东西，然后spring会在系统运行到适当的时候，把你要的东西主动给你，同时也把你交给其他需要你的东西。所有的类的创建、销毁都由 spring来控制，也就是说控制对象生存周期的不再是引用它的对象，而是spring。对于某个具体的对象而言，以前是它控制其他对象，现在是所有对象都被spring控制，所以这叫控制反转。

​ DI—Dependency Injection，即“依赖注入”：组件之间依赖关系由容器在运行期决定，形象的说，即由容器动态的将某个依赖关系注入到组件之中。依赖注入的目的并非为软件系统带来更多功能，而是为了提升组件重用的频率，并为系统搭建一个灵活、可扩展的平台。通过依赖注入机制，我们只需要通过简单的配置，而无需任何代码就可指定目标需要的资源，完成自身的业务逻辑，而不需要关心具体的资源来自何处，由谁实现。

​ AOP即切面编程，把核心业务和非核心业务进行分离，即专人做专事，用于降低耦合度，提高代码的复用性，将非核心业务做为一个切面，在spring中即将bean进行强化和包装，将部分业务捆绑到bean中。

BeanFactory API

​ beanfactory是spring中的底层api，是bean工厂，用于创建bean，beanfactory的开发步骤是程序代码调用第三方，给出配置清单，第三方去获取程序需要的资源，即配置bean。

  1. 导入spring的jar包或者由maven创建
  2. 定义接口和实现类
  3. 创建bean.xml文件，将实现类信息配置到xml
  4. 测试代码，创建beanfactory，加载配置文件，获取对象

通过这种方式成功装配到bean，所需要的bean需要在xml中声明，在程序代码中通过getbean调用注入

Spring的本质是一个bean工厂(beanFactory)或者说bean容器，它按照我们的要求，生产我们需要的各种各样的bean，提供给我们使用。beanfactory可以管理bean，对bean进行一些操作。

ApplicationContext API

​ 内部封装了beanfactory，比beanfactory更丰富和强大。

ApplicationContext 直接通过ClassPathXmlApplicationContext去解析xml文件，然后直接调用getbean方法。

BeanFactory和ApplicationContext的关系

  1. beanfactory是早期接口，applicationcontext是后期更高级的接口，称之为spring容器
  2. applicationcontext对beanfactory进行了扩展，beanfactory api更偏向于底层
  3. applicationcontext继承了beanfactory。
  4. bean的初始化时机不同，beanfactory是首次调用getbean是才进行bean的创建，而applicationcontext则在配置文件加载，容器一创建就将bean实例化并初始化好。

bean的范围配置

默认情况下，单纯的spring环境中bean的作用范围有两个：singleton和prototype

  • singleton:单例，默认值，在spring容器创建时就进行bean的实例化，存储到容器的单例池，每次getbean但是从单例池中获取相同的bean实例。获取到的bean都是一样的实例。
  • prototype：原型，spring容器初始化不会创建bean实例，当调用getbean时才会实例化bean，每次getbean都会创建一个新的bean实例。获取到的bean每次都不一样。

如果只是基本的spring环境，scope值只有singleton、prototype，如果是有spring-webmvc环境的话其实还有request、session。

Spring后处理器

后处理器是Spring对外开放的重要扩展点，允许我们介入到Bean的实例化流程中，以达到动态注册BeanDefinition，动态修改BeanDefinition，以及动态修改Bean的作用，Spring有两个后处理：

  • BeanFactoryPostProcessor：Bean工厂后处理器，在BeanDefinitionMap填充完毕之后，在Bean实例化之前执行；
  • BeanPostProcessor：Bean后处理器，一般在bean实例化之后，填充到单例池之前。

BeanFactoryPostProcessor

理解一下呢，在bean配置之后，会将这些放到BeanDefinitionMap中，之后调用BeanDefinitionRegistryPostProcessor进行注册添加，再通过BeanFactoryPostProcessor后处理器，最后实例化bean到单例池中。

其实后面的注解开发，实例化bean多半也是利用beanfactory的后处理器进行注册的

SpringBean的生命周期

​ 通过ClassDefinitionApplication读取到xml文件之后，BeanDefinition将xml中配置的bean装配到BeanDefinitionMap中，这时候再通过beanfactory后处理器进行处理，这一步位于map装配之后，bean实例化之前，在bean实例化后，这时候是一个半成品，进行bean后处理器的加工增强，装配方法和其他bean，bean后处理器作用于bean实例化之后，填充单例池之前，最后进行单例池的填充，就可以通过getbean去调用bean了
